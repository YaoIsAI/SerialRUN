# SerialRUN 功能评估报告

## 1. 多串口同时监控

### 当前状态
- ❌ 不支持多串口同时监控
- 当前架构：单一 port_owner 管理一个串口
- CAN/Scope/FileTransfer 需要独占串口

### 实现方案

**方案 A：多 port_owner 实例**
```
AppState {
    port_owners: HashMap<String, PortOwnerHandle>,  // 按端口名索引
    active_port: Option<String>,                      // 当前活动端口
}
```
- 优点：每个端口独立管理，互不影响
- 缺点：内存占用增加，需要 UI 适配多端口选择

**方案 B：共享 port_owner + 端口切换**
```
AppState {
    port_owner: Option<PortOwnerHandle>,
    port_configs: Vec<SerialConfig>,  // 保存多个端口配置
}
```
- 优点：内存占用小，UI 改动小
- 缺点：同一时间只能操作一个端口

### 推荐方案
**方案 A**（多 port_owner 实例），因为：
1. 用户可能需要同时监控多个设备
2. Modbus 主站可能需要连接多个从站
3. TCP 桥接需要同时访问多个串口

### 实现复杂度
- 高（需要修改 port_owner 架构、UI 布局、所有面板）
- 预计工作量：2-3 周

## 2. 在线更新功能

### 当前状态
- ❌ 不支持在线更新
- 用户需要手动下载新版本

### 实现方案

**方案 A：GitHub Release 检查**
```
1. 启动时检查 GitHub API
2. 比较版本号
3. 下载新版本到临时目录
4. 提示用户重启更新
```
- 优点：简单，利用现有 GitHub 基础设施
- 缺点：需要 GitHub 访问权限

**方案 B：自建更新服务器**
```
1. 部署更新服务器（如 S3 + CloudFront）
2. 客户端定期检查
3. 下载差量更新（bsdiff）
4. 自动替换可执行文件
```
- 优点：可控，支持差量更新
- 缺点：需要维护服务器

**方案 C：使用 update-rs 库**
```
// Cargo.toml
[dependencies]
self_update = "0.30"
```
- 优点：成熟的库，支持多种更新方式
- 缺点：增加依赖

### 推荐方案
**方案 C**（使用 self_update 库），因为：
1. 成熟稳定，社区维护
2. 支持 GitHub Release、S3 等多种源
3. 自动处理下载、验证、替换
4. 工作量小（约 1-2 天）

### 实现步骤
1. 添加 `self_update` 依赖
2. 在 main.rs 启动时检查更新
3. 在帮助菜单添加"检查更新"按钮
4. 配置 GitHub Release 作为更新源

### 实现复杂度
- 低（使用成熟库）
- 预计工作量：1-2 天
